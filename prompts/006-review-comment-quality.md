<requirements>
Thoroughly analyze every comment in the codebase. Remove or rewrite comments that match ANY of these categories:

1. **AI-style comments** — Overly formal, explanatory comments that sound like documentation generated by an AI:
   - "This method handles the processing of..."
   - "This function is responsible for..."
   - "NOTE: This was added to fix..."
   - "ARCHITECTURE FIX: ..."
   - Unnecessarily verbose docstrings that restate the function signature

2. **Obvious/redundant comments** — Comments that state what the code already says:
   - `self.store = store  # Store the store reference`
   - `return result  # Return the result`
   - `# Initialize the list` above `items = []`
   - `# Call the function` above a function call

3. **Deprecated/stale comments** — References to old code, removed features, or past decisions:
   - "# Legacy compatibility" on current code
   - "# DEPRECATED - use X instead" where the deprecated thing is already gone
   - "# TODO: remove this" for code that should just be removed
   - "# Backward compat" wrappers that no longer wrap anything
   - "# OLD: ..." or "# Previously: ..."
   - References to removed classes, methods, or modules

4. **Non-production comments** — Debug artifacts or development notes:
   - `# print(...)` or commented-out code
   - `# DEBUG`, `# TEMP`, `# HACK`, `# FIXME` without actionable context
   - `# For testing only`
   - Commented-out import statements

5. **Commit-message-style comments** — Comments that describe what changed rather than why the code exists:
   - "# Added in v2.1"
   - "# Refactored from old_module.py"
   - "# Moved from MainWindow"
   - "# HIGH-003 FIX: ..."
   - "# MW-04 fix"
   - Bug/ticket references in inline comments (these belong in git history, not code)

6. **Over-documented sections** — Section separators or headers that add no value:
   - `# === Data Operations ===` (if the class structure is already clear)
   - `# ============` banner comments in test files (keep only if they genuinely help navigation in very long files)

Do NOT remove:
- Comments explaining non-obvious business logic or domain rules
- Comments explaining WHY a workaround exists (not just WHAT it does)
- Type: ignore comments with justification
- License headers
- Genuine TODO comments with actionable descriptions
- Docstrings that add real value beyond the function signature (e.g., explaining edge cases, parameter constraints, return value semantics)
</requirements>
